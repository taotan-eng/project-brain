# Evaluation Report Template

This file is the **canonical shape** for every daily evaluation report
generated by a handoff's End-of-day evaluation script. The underscore
prefix marks it as a template (so the daily report files at
`day-NN-evaluation-report.md` sort separately and the template is never
mistaken for an actual report).

## When this template is used

- Every handoff doc (`day-NN-<slug>.md`) includes an `## End-of-day evaluation` section containing a bash script.
- That script evaluates the handoff's PR-merge criteria and writes a markdown file at `docs/handoff/day-NN-evaluation-report.md`.
- The written file follows the structure below. The handoff's bash script is a parameterized copy of the skeleton at the bottom of this file.

## Required structure

Every evaluation report must contain, in order:

1. **H1 title**: `# Day-NN Evaluation Report` (no other H1 in the file).
2. **Metadata block** (bulleted): generated timestamp (ISO 8601 UTC), plan reference (path + section), handoff reference (path), and any predecessor-report inputs that are relevant (e.g., the prior day's audit report).
3. **Optional § "Script adjustments from handoff spec"**: if the executor adjusted any criterion check from the handoff spec, document the adjustment + rationale here. Adjustments must be transparent; CC's day-1 evaluation set the example by listing two adjustments (criterion 1's `tail -3` and criterion 5's baseline-error exclusion) with reasoning.
4. **§ Merge criteria**: a markdown table with columns `# / Criterion / Status / Evidence`. One row per criterion, one ✓ or ✗ per row, evidence is programmatic output (file sizes, counts, exit codes, validator tail lines). All criteria must appear in the table even if they pass — the table is the at-a-glance pass/fail summary.
5. **§ Files changed**: a code block containing `git diff --stat` output scoped to this day's changes (either `git diff --stat <start-commit>..HEAD` or, if no clear start commit, `git diff --stat HEAD~N..HEAD` where N covers the day's commits).
6. **§ Commits** (sometimes titled "Commits (most recent N)"): a code block containing `git log --oneline` output covering this day's commits.
7. **§ Verdict**: an H2 heading with one of two exact values: `## Verdict: **MERGE-READY**` or `## Verdict: **NOT-READY**`. Followed by a one-paragraph summary. If `NOT-READY`, the paragraph names the failing criterion (or criteria) and references the failure modes section of the handoff.

Optional but encouraged:

- **§ Day-N+1 gating notes**: anything the executor wants to flag for the next handoff (e.g., scope rebaseline needed, escalation thresholds breached). Day-1's report demonstrated this with the audit-count escalation note.

## Filename convention

`docs/handoff/day-NN-evaluation-report.md` — zero-padded two-digit day number,
no other suffix. The file is committed as part of the exit-gate sequence
and used as the PR body via `gh pr create --body-file …`.

## What "MERGE-READY" means

All PR-merge criteria listed in the handoff's `## PR-merge criteria` section pass their programmatic checks. The verdict is determined by ANDing the per-row results; a single ✗ flips the whole report to `NOT-READY`.

The executor does NOT use judgment to override a failed criterion. If a criterion check is misguided (the day-1 `tail -3` case proved this happens), the correct response is:

1. Document the adjustment in § "Script adjustments from handoff spec" with the specific reasoning.
2. The adjusted check counts as the actual programmatic check for that criterion.
3. If the adjustment can be justified post-hoc, the verdict can be MERGE-READY.
4. The next handoff fixes the criterion definition so the spec matches the working check (day-2 inherits the `tail -3` → whole-file-grep fix because day-1's adjustment was correct).

This means a reviewer reading the PR body can always check whether an adjustment was made and whether it was justified — transparency, not hand-waving.

## Skeleton bash script

The script below is the canonical skeleton. Each handoff customizes it by
filling in the per-criterion blocks. The structural pieces (criteria-table
build, files-changed, commits, verdict line, exit code) stay identical
across handoffs to keep reports consistent.

```bash
set +e   # evaluate every criterion; don't bail on the first failure
cd /Users/ttan/workspace/Project-Brain/final/project-brain

REPORT=docs/handoff/day-NN-evaluation-report.md
PASS=1
TIMESTAMP=$(date -u +%Y-%m-%dT%H:%M:%SZ)
declare -a ROWS=()

# ------------------------------------------------------------
# Per-criterion blocks — customize for this handoff
# ------------------------------------------------------------

# Criterion 1 — <name>
if <programmatic check>; then
  ROWS+=("| 1 | <name> | ✓ | <evidence string> |")
else
  ROWS+=("| 1 | <name> | ✗ | <what failed> |")
  PASS=0
fi

# Criterion 2 — ...
# Criterion 3 — ...
# (...)

# ------------------------------------------------------------
# Build the report — KEEP THIS BLOCK STABLE ACROSS HANDOFFS
# ------------------------------------------------------------

{
  echo "# Day-NN Evaluation Report"
  echo
  echo "- Generated: ${TIMESTAMP}"
  echo "- Plan reference: \`<path-to-plan-artifact>\` § <section>"
  echo "- Handoff: \`docs/handoff/day-NN-<slug>.md\`"
  # Optional input-report references:
  # echo "- Day-N-1 input: \`docs/handoff/day-N-1-<something>.md\`"
  echo
  # If any adjustments to the handoff's criterion checks were made,
  # document them here under "## Script adjustments from handoff spec":
  # echo "## Script adjustments from handoff spec"
  # echo
  # echo "- **Criterion X:** ..."
  # echo
  echo "## Merge criteria"
  echo
  echo "| # | Criterion | Status | Evidence |"
  echo "|---|---|---|---|"
  for row in "${ROWS[@]}"; do echo "$row"; done
  echo
  echo "## Files changed"
  echo
  echo '```'
  # Pick the right --stat range for this day's commits. Either:
  #   git diff --stat <start-of-day-commit>..HEAD
  # or as a heuristic:
  #   git diff --stat HEAD~<N>..HEAD   # where N covers this day's commits
  (git diff --stat HEAD~5..HEAD 2>/dev/null || git diff --stat HEAD) | sed 's/^/  /'
  echo '```'
  echo
  echo "## Commits"
  echo
  echo '```'
  git log --oneline -10 | sed 's/^/  /'
  echo '```'
  echo
  if [ "$PASS" = "1" ]; then
    echo "## Verdict: **MERGE-READY**"
    echo
    echo "All <N> criteria pass. This day's work is ready to merge."
  else
    FAILED=$(printf '%s\n' "${ROWS[@]}" | grep "✗" | awk -F'|' '{print $2}' | tr -d ' ')
    echo "## Verdict: **NOT-READY**"
    echo
    echo "Failing criteria: ${FAILED}. See merge-criteria table above for evidence. Re-engage the relevant task's test-fix-retest loop, or escalate if the failure is judgment-bound (see § Escalation conditions in the handoff)."
  fi
} > "$REPORT"

cat "$REPORT"
echo
if [ "$PASS" = "1" ]; then
  echo "✓ DAY-NN EVALUATION: MERGE-READY — report at $REPORT"
  exit 0
else
  echo "✗ DAY-NN EVALUATION: NOT-READY — report at $REPORT"
  exit 1
fi
```

## Anti-patterns

These are explicit "do not do" rules learned from day-1:

- **Do not collapse multiple criteria into a single row.** Each criterion gets its own row, its own evidence cell, and its own ✓/✗. Bundling makes it impossible to tell which gate failed.
- **Do not write "passes" as the evidence cell.** Evidence is the actual programmatic output (number, file size, validator tail line). "Passes" tells the reviewer nothing.
- **Do not skip § Files changed or § Commits.** They make the PR body self-contained — the reviewer doesn't have to switch tabs to see what was done. Keep them even on small-diff days.
- **Do not put the verdict line in any format other than `## Verdict: **MERGE-READY**` or `## Verdict: **NOT-READY**`.** Future automation (CI, dashboards, status pages) will grep for these exact strings.
- **Do not silently exclude a criterion.** If you can't evaluate one, write a row for it with status `?` and evidence explaining why, and set `PASS=0` (treat un-evaluable as fail-safe). Better to show a `?` than to leak a criterion entirely.
- **Do not declare MERGE-READY when any criterion is ✗ or `?`.** The verdict mechanic is `MERGE-READY iff all rows are ✓`. There is no override path.
- **Do not produce the report and then proceed to push/PR if it's NOT-READY.** The handoff's exit gate is gated on the report's verdict; NOT-READY routes back to the test-fix-retest loop, not forward to `gh pr create`.
